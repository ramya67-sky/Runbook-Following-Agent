# VEGA — Runbook Automation & Execution Agent
# Vega defines 

 V-Verify
 
 E-Execute
 
 G-Guide
 
 A-Audit
 

AI-powered runbook execution engine with risk classification, human-in-the-loop approval, and automatic error remediation.

---

## Overview

VEGA is a full-stack DevOps automation platform that parses, classifies, and executes IT runbooks step by step. It combines a Python/Flask backend with a React frontend to give operators a supervised, auditable pipeline for running operational procedures — with AI-assisted command classification, multi-step approval flows, and built-in error recovery.

When a runbook step fails, VEGA automatically invokes an MCP (Model Context Protocol) server to suggest a corrected command and retries the step — without operator intervention for low-risk operations.

---

## Features

- **Runbook parsing** — Upload a Markdown runbook or paste it directly. VEGA extracts numbered steps and identifies executable commands, including NLP-inferred commands from natural language descriptions (powered by Ollama).
- **Multi-type step execution** — Supports four execution modes: `SHELL`, `REST_API`, `DB_QUERY` (SQL), and `CLOUD_CLI` (aws, gcloud, kubectl, terraform, etc.).
- **AI risk classification** — Each command is classified as `SAFE`, `MEDIUM`, or `HIGH` risk using a local Ollama LLM, with a deterministic rule-based fallback when Ollama is unavailable.
- **Human-in-the-loop approval** — Medium and high-risk steps pause execution and require explicit operator approval or denial before proceeding.
- **Command allowlist enforcement** — A configurable allowlist of safe base commands provides a hard security boundary independent of AI classification.
- **Auto-remediation via MCP** — When a safe step fails, the MCP client connects to a local `mcp_server.py` to retrieve a corrected command and automatically retries.
- **Discord notifications** — Real-time webhook notifications for run start, step results, approval requests, and run completion. Messages are generated by Ollama for human-readable context.
- **Audit reports** — Each completed run generates a structured audit summary covering total steps, execution counts, errors, skipped steps, and duration.
- **Run history & metrics** — A persistent SQLite (or MySQL) database stores all runs and steps, with aggregate stats (completed, failed, partial) available via API.

## Getting Started

### Prerequisites

- Python 3.10+
- Node.js 18+
- [Ollama](https://ollama.ai) installed and running locally (optional — app falls back to rule-based classification if unavailable)

### Backend Setup

```bash
cd backend

# Create and activate a virtual environment
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Start the Flask server
python app.py
```

The backend runs on `http://localhost:5050` by default.

### Frontend Setup

```bash
cd frontend

# Install dependencies
npm install

# Start the development server
npm run dev
```

The frontend runs on `http://localhost:5173` by default.

### Ollama Setup (Optional)

```bash
# Pull the default model
ollama pull llama3

# Ollama serves on http://localhost:11434 by default
ollama serve
```

If Ollama is not running, VEGA automatically enables `MOCK_OLLAMA` mode and uses deterministic rule-based classification and parsing.

## Configuration

All configuration is managed via environment variables or `backend/config.py`.

| Variable | Default | Description |
|---|---|---|
| `OLLAMA_API_URL` | `http://localhost:11434` | Ollama API endpoint |
| `OLLAMA_MODEL` | `llama3` | Model name to use for classification and parsing |
| `MOCK_OLLAMA` | `false` | Force rule-based fallback (no Ollama calls) |
| `DATABASE_URL` | SQLite (local file) | SQLAlchemy connection string; set to `mysql+pymysql://...` for MySQL |
| `DISCORD_WEBHOOK_URL` | _(empty)_ | Discord webhook URL for notifications; leave blank to disable |
| `SECRET_KEY` | _(default)_ | Flask session secret key — **change this in production** |

### Command Allowlist

The allowlist in `config.py` defines which shell binaries may execute without triggering a high-risk block:

```python
COMMAND_ALLOWLIST = {
    "df", "ls", "cat", "grep", "journalctl", "systemctl",
    "ps", "ping", "curl", "echo", "uptime", "free",
    "whoami", "pwd", "head", "tail"
}
```

Cloud CLI tools (`aws`, `gcloud`, `az`, `kubectl`, `terraform`, `helm`) are detected separately and always require manual approval regardless of sub-command.

## API Reference

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/runs` | List all runs with aggregate stats |
| `POST` | `/api/upload` | Upload or submit a runbook for staging |
| `GET` | `/api/run/<id>/status` | Get full status and steps for a run |
| `POST` | `/api/run/<id>/execute_next` | Execute the next pending step |
| `POST` | `/api/run/<id>/cancel` | Cancel a run and mark remaining steps as denied |
| `POST` | `/api/step/<id>/approve` | Approve and execute a step pending human review |
| `POST` | `/api/step/<id>/deny` | Deny execution of a pending step |
| `POST` | `/api/runs/clear` | Wipe all run history from the database |

## Runbook Format

VEGA accepts Markdown runbooks with numbered steps. Commands should be wrapped in backticks. If no backtick command is present, Ollama will infer one from the step description.

```markdown
# Service Recovery Runbook

1. Check available disk space: `df -h`
2. Verify the application service is running: `systemctl status myapp`
3. Review the last 100 lines of logs: `journalctl -u myapp -n 100`
4. Restart the application service: `systemctl restart myapp`
5. Confirm the service came back up: `systemctl status myapp`
```

SQL steps are prefixed with `SQL:`:

```markdown
6. Check for stuck jobs: `SQL: SELECT * FROM jobs WHERE status = 'RUNNING' AND started_at < NOW() - INTERVAL 1 HOUR;`
```

REST API steps use HTTP method syntax:

```markdown
7. Health check the upstream API: `GET https://api.example.com/health`
```

## Step Lifecycle

```
PENDING → RUNNING → SUCCESS
                  ↘ FAILED → RECOVERING → SUCCESS / FAILED
       ↘ WAITING_APPROVAL → (approved) → RUNNING
                           → (denied)  → DENIED
```

## Security Considerations

- The command allowlist is the primary hard boundary. Commands outside it are blocked or require approval regardless of AI classification.
- Cloud CLI commands (`kubectl`, `terraform`, etc.) always require explicit operator approval.
- `HIGH` risk commands (destructive operations such as `rm -rf`, `DROP TABLE`, `shutdown`) are never auto-executed.
- The `SECRET_KEY` in `config.py` should be replaced with a strong random value before any non-local deployment.
- This project is designed for trusted internal/operator use. Do not expose the Flask API to untrusted networks without adding authentication.
